KidBytes · Beacon-Locator Puzzle Box

Conceptual / Functional Architecture — FRDM-MCXA153 · HM-10 · PC Companion  ·  SRS 9.40 / 10  ·  FRs F1–F8 (39)

How to read this diagram

Each of the 11 zones is explained below in the format purpose → depends on (cross-layer) → depends on (intra-layer) → measured number . The intra-layer line names which modules in the same zone depend on each other — read it to see what can be built and tested in parallel vs. what has a forced ordering. LVGL is out: MCXA153 has only 36 KiB SRAM (24 + 8 + 4). The real view code (Layer_View_SDK.zip) drives a TFT ILI9341 240×320 RGB565 panel over LPSPI with a chunked partial framebuffer. Sources: Product Report §3.1.2 traceability matrix, SRS IEEE 830 9.40/10, MCXA153 Data Sheet, Layer_View_SDK — hal/display.h, drivers/display_drv.c, v_*.c.

1 ENVIRONMENT — external actors

Purpose. Everything that exists outside the MCU boundary and generates or receives events. Kids, Admin PC, 5 iBeacons in 5 rooms, reward chamber, and the silicon constraints of the MCU itself.

Depends on (cross-layer). Nothing — this layer is the world. Downstream depends on it.

Depends on (intra-layer). The five actors are independent of each other in normal operation. Two coupling rules: (a) Reward chamber only opens when the kid finishes the route across all five iBeacons (functional sequence, not a build-time dep); (b) the MCU constraints row is the silicon ceiling that bounds what every other layer can do, so it is read-only context for the rest of the diagram.

Measured numbers.

  • Kid press rate: ~1 press / 2 s (aperiodic)
  • iBeacon advert period: 100 ms
  • MCU: Cortex-M33 · 96 MHz · 128 KiB flash · 36 KiB SRAM (24+8+4)
  • Silicon facts: NO_FPU, no MPU, no SAU, 3 priority bits → 8 exception priorities
2 SYSTEM INPUTS — MCU boundary

Purpose. Everything that crosses into the MCU — buttons, BLE UART, USB-CDC, power bank — plus the explicit timing quality goals this boundary must respect.

Depends on (cross-layer). Environment (1) generates the events; HAL (3) defines how they are read.

Depends on (intra-layer). The four input ports are mutually independent at the electrical level (separate pins, separate peripherals: GPIO3 / LPUART0 / USB / VDD). The Quality goals (timing) row is shared by all four — it is a constraint over the whole layer, not a fifth peer. Concretely: button ISR latency competes for CPU with BLE UART RX, so the worst-case sum (10 + 500 + 80 + 20 = 610 ms) must fit a 1 s scan-task period.

3 HAL — portable headers

Purpose. Seven .h files (button.h · ble.h · usb.h · display.h · lock.h · nvm.h · feedback.h) that define what the firmware can ask hardware to do, independent of vendor.

Depends on (cross-layer). Nothing — pure headers. Downstream drivers implement them; upstream View, Functional and Model use them.

Depends on (intra-layer). The seven HAL headers are fully independent (zero #include between them, only <stdint.h> / <stdbool.h>). One narrow exception: feedback.h shares the LED on PORT3/GPIO3 with button.h wiring, so pin-mux conflicts must be resolved at the board layer — not in HAL.

Measured number. Each HAL header has 0 #ifdef blocks; verified against Layer_View_SDK/hal/display.h (only <stdint.h>, no MCU-specific include).

4 DRIVERS — MCXA153-specific

Purpose. Implement each HAL header using one real NXP MCUXpresso SDK call per peripheral. One driver = open / close / read / write / ioctl.

Depends on (cross-layer). HAL headers (3), NXP MCUXpresso SDK, physical peripherals (System Inputs/Outputs).

Depends on (intra-layer). Each driver wraps a different peripheral instance (LPUART0 / LPI2C1 / eFlexPWM0 / flash / GPIO3 / CTIMER / USB), so there is no driver-to-driver C-API call. Two clock-domain dependencies are real and shared at MRCC: every driver assumes FRO_HF → 96 MHz core / 48 MHz peripheral bus is up. display_drv in Layer_View_SDK uses tft_lcd + lpspi_master as a sub-driver — a legitimate intra-layer dependency since both belong to Layer 4.

5 FUNCTIONAL — Active Object

Purpose. Single controller — main_fsm — that owns every state change. Tasks (scan_task, logger_task, event_dispatcher) only hand events to it through a 16-slot priority queue. No shared state, no locks needed.

Depends on (cross-layer). Drivers (4) for incoming events, FreeRTOS task API for scheduling, Model (7) (config_store, location_engine, puzzle_engine) for decisions, View (6) for output cues.

Depends on (intra-layer). Strictly one-direction: scan_task → event_dispatcher → main_fsm → logger_task. The dispatcher is the only thing the FSM blocks on; tasks never call each other directly. The two FSM super-states (ADMIN, FAULT) inherit from the same dispatcher queue, so adding a new event to the FSM never adds a new producer task. Removable: logger_task can be disabled with LogGlobalOff() and the FSM still runs.

States. IDLE → LOCATOR → PUZZLE → UNLOCK_NEXT → FINAL, plus two super-states ADMIN (USB) and FAULT (BLE_LOST, NVM_FAIL, BROWNOUT, ASSERT).

Measured numbers. scan_task period = 250 ms (F1.1), dispatcher queue = 16 slots, log flush ≤ 20 ms.

6 VIEW — raw framebuffer (no LVGL)

Purpose. Render to the TFT ILI9341 panel and the PC companion. LVGL is not used — its minimum memory footprint (~40 KiB RAM at its smallest configuration) exceeds the MCXA153's entire 36 KiB SRAM split across 3 banks. Instead, v_screen writes characters directly through display_put_string, and v_icons uses a small chunked region of lcd_framebuffer with lcd_set_area + lcd_write_pixels.

Depends on (cross-layer). display_drv (Layer 4) for screen output, feedback_drv (Layer 4) for LED + buzzer (today bypassed — v_fb.c writes MRCC0/PORT3/GPIO3 directly, see leak #3 in the dependency doc), usb_drv (Layer 4) for the PC link, and the Functional FSM (5) for state-driven calls.

Depends on (intra-layer). The three on-MCU views (V_SCREEN, V_ICONS, V_FB) all share the same TFT panel — they coexist by writing to disjoint screen regions: V_SCREEN draws Room/Next/Puzzle text rows at y = 0, 2h, 4h; V_ICONS draws RSSI bars top-right and battery/fault bottom corners. No view ever reads back what another wrote, so the order of calls within one frame does not matter. V_PC is on a different transport (USB-CDC) and does not touch the TFT path at all.

7 MODEL — host-testable

Purpose. Pure C modules that do the actual problem-solving. No MCU calls, no fsl_*. Can be unit-tested on the PC with a normal C compiler.

Depends on (cross-layer). Nothing at build time except <stdint.h>/<string.h>. At run time the Functional FSM (5) calls them with commands and reads results.

Depends on (intra-layer). Two real intra-layer dependencies: (a) config_store is read by location_engine (room map), puzzle_engine (answers), route_manager (sequence) and log_model (admin secret) — it is the shared, read-mostly source of truth. (b) route_manager consumes location_engine's room id to advance its k-of-5 progress. log_model is a sink only — nothing depends on it. None of the five modules has compile-time #include of another — dependencies are passed by function arguments, so each can be unit-tested standalone.

8 SYSTEM OUTPUTS — actuators + log

Purpose. Physical and logical effects the kid (or the admin) sees.

Depends on (cross-layer). Drivers (4) that push bytes to pins; Model (7) (log_model) that produces the USB log stream; Mechanical (10) for the lock drive latch.

Depends on (intra-layer). The four outputs are physically independent — separate pins, separate peripherals — and can be exercised one at a time. Two soft couplings exist by sequence, not by wire: (a) the Buzzer / LED cue accompanies a successful puzzle and again at the unlock event, so they fire in pairs of frames; (b) the Electronic lock opening at FINAL is the only output that should also produce a USB log CRIT line, so loss of one is detectable from the other.

  • Electronic lock — closed by default, opens at FINAL (F4.1, F4.2)
  • Display frames — refresh ≤ 1 s for iconography (F3.3)
  • Buzzer / LED — puzzle and unlock cues (F3.2, F4.3)
  • USB log stream — INFO · DEBUG · WARN · ERROR · CRIT (F8)
9 POWER + WATCHDOG — orthogonal

Purpose. Cross-cutting. Runs alongside main_fsm, never inside it. Governs MRCC clock gates + WWDT.

Depends on (cross-layer). System Inputs (2) for the power-bank rail; HAL/Drivers (3, 4) only as observers (no driver gates clocks for itself); Functional FSM (5) only as a watchdog kicker, not a controller.

Depends on (intra-layer). The three sub-modules form a dominance chain: PWR measures the rail → pwr_fsm chooses Active/Sleep/Deep-Sleep → WWDT only kicks in the Active state. If PWR reports brown-out, pwr_fsm forces Deep-Sleep and the WWDT is held off (per the data sheet). The chain is orthogonal to main_fsm: main_fsm never reads or writes power state directly.

  • PWR — battery + brown-out + LP / deep-sleep (F7.1, F7.4)
  • pwr_fsm — orthogonal: Active · Sleep · Deep-Sleep
  • WWDT — kick ≤ 100 ms else reset
10 MECHANICAL

Purpose. The physical box — lock + hinge + child-safe enclosure + reward drive latch (F4, F7.1). Not firmware, but the lock and enclosure dimensions constrain the PWM pulse window of lock_drv.

Depends on (cross-layer). System Outputs (8): mechanical only moves when Electronic lock is energised. Drivers (4): lock_drv sets the PWM duty that the servo translates to a latch travel.

Depends on (intra-layer). Single module — no intra-layer dependency. The enclosure and lock share screws/holes, but that is a CAD-level concern, not a firmware one.

Measured number. Latch travel time ≤ 500 ms; servo PWM pulse window 1 – 2 ms at 50 Hz (standard hobby servo). Holds the report's F4 budget.

11 LOG — cross-cutting (F8)

Purpose. Single log.h API used by every driver, the FSM, and the model. printf is banned — too heavy and not per-subsystem gated.

Depends on (cross-layer). usb_drv (Layer 4) as the byte sink — logger_task empties the ring into USB-CDC. Every other layer is a caller of log.h; the LOG layer never calls them back.

Depends on (intra-layer). One file, one ring buffer, one mutex. The sys tag and level filter are independent dimensions — turning off all DISPLAY messages does not affect POWER messages, and dropping DEBUG globally does not silence CRIT. Concretely: LogSetOutputLevel(DISPLAY, NONE) is local; LogGlobalOff() is global.

API.

Log(sys, level, msg);
LogWithNum(sys, level, msg, n);
LogSetOutputLevel(sys, level);
LogGlobalOn();
LogGlobalOff();

sys   ∈  { SYS, COMMS, DISPLAY, SENSOR, FSM, POWER }
level ∈  { NONE, INFO, DEBUG, WARN, ERROR, CRIT }
